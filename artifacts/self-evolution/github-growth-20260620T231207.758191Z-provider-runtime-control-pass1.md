## Hypothesis

Provider-runtime-control windows need a current-action-level preflight panel.
The existing completion gate reported missing, degraded, blocked, or passed
provider-runtime samples near slice completion, but pass-1 supervisors still had
to correlate that with `current_action` manually. A top-level body-free panel
should make the next recovery step visible without exporting provider bodies,
raw URLs, raw diagnostics, or runtime values.

## Change

- Added `current_action_provider_runtime_preflight` to
  `skill_route_discovery_lane` output.
- The panel reports the selected bounded lane, pass numbers, selected item IDs,
  provider replay sample status, recovery hint codes and hashes, and replay
  commands.
- Missing samples block with `provider_runtime_preflight_sample_missing`;
  blocked samples require recovery-hint replay; degraded samples are reviewable
  but cannot claim success; passed samples mark the current action ready.
- Runtime action, provider launch, remote execution, external skill activation,
  raw preflight inputs, raw diagnostics, raw provider values, raw source URLs,
  raw target paths, and upstream bodies remain denied.
