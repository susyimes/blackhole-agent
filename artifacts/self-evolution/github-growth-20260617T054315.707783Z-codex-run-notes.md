## Evidence

- `https://github.com/omnigent-ai/omnigent/pull/453` motivated hardening pending-card handling so unrelated pending state does not contaminate the current user-visible status.
- The local repository does not have Omnigent browser-card or file-upload e2e machinery. The closest executable surface is supervisor child-session liveness, which already tracks pending child elicitations and user-visible runner status.

## Hypothesis

If heartbeat child-session records include pass ownership metadata, the supervisor should ignore records owned by another wake pass. Otherwise a stale pending child prompt from an older pass can make the current runner look active and block clean asleep/status reporting.

## Changes

- Added pass ownership filtering in `supervisor_child_summary_from_heartbeat`.
- Preserved backward compatibility for child sessions without pass/run ownership metadata.
- Ignored aggregate `active_child_count` overrides when owner-tagged child session metadata is present, so stale owner-tagged pending cards cannot override the filtered current-pass view.
- Added a regression test for stale pending child metadata from another pass.
- Left `docs/self-model.md` unchanged because its current preference for rollback-backed, validated local evolution matched this run.
