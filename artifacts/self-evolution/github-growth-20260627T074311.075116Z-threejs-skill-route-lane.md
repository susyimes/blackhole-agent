## Hypothesis

The repeated Three.js game-skill fork cluster is useful as local route evidence,
but repeated forks should not increase activation readiness. The safest local
improvement is an operator-visible harness lane that collapses the cluster into
one `game_frontend_workflow` profile, selects the bounded local `test` lane, and
keeps upstream install, scaffold execution, provider launch, asset generation,
and external skill activation denied.

## Change

skill-route harness output, plus a replay fixture and focused test coverage.

The self-model was read and left unchanged. It already describes the relevant
preference for rollback-backed local evolution while keeping external skill
activation outside discovery.
