## Evidence

Omnigent PR #881 adds an on-demand Kubernetes managed-sandbox provider. The useful local lesson is not to launch a cluster workload from trend evidence, but to fail closed before launch when required Kubernetes provider metadata is missing or malformed, when credential-looking env keys are inlined into pod config, or when launch-token handling is not explicitly server-minted.

## Hypothesis

`provider_runtime_preflight` should make Kubernetes sandbox-provider readiness visible as body-free diagnostics. A managed runner path should be locally replayable without cluster access, while raw namespace, image, service-account, secret, selector, env-key, token, provider config, and cluster-response values remain omitted.

## Changes

- Added `kubernetes_sandbox` metadata validation inside `provider_runtime_preflight`.
- Added fail-closed recovery classes for missing config, malformed Kubernetes names/selectors/image metadata, inline credential env keys, missing server-minted launch-token metadata, and inline launch-token values.
- Added operator recovery hints that export only booleans and counts.
- Documented the Kubernetes sandbox-provider preflight contract in `docs/architecture.md`.
- Added focused regression coverage in `tests/test_harness_eval.py` and `tests/test_docs_contracts.py`.
