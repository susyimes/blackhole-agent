## Hypothesis

The previous provider-runtime passes added local preflight diagnostics for startup,
proxy/env, and runner turn-context failures. The final pass should make the
operator-facing completion report state how provider/runtime wording in
skill-route evidence is interpreted: body-free diagnostics and recovery hints,
then local replay, then bounded documentation, config, test, or code_patch
validation. It should not create a provider-runtime activation lane.

## Change

- Added `provider_runtime_interpretation_panel` to the pass-4
  `completion_report`.
- The panel summarizes diagnostic panel status, sample gate status, recovery
  summary readiness, completion handoff status, recovery hint codes, route
  profile hashes, and replay commands.
- The panel repeats denials for runtime action, external skill activation,
  external harness execution, provider launch, remote execution, raw evidence
  URLs, raw source URLs, raw preflight inputs, raw diagnostics, raw provider
  values, raw target paths, and upstream bodies.
- Documented the interpretation rule in `docs/skill-route-discovery.md`.
