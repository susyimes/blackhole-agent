## Hypothesis

The existing route-hint policy validates configured route lanes, but a malformed selected item could carry a pre-shaped
`route_classification.allowed_lanes` with an unsupported lane such as `runtime_execution`. The operator-facing
`skill_route_local_lane_candidates` panel filtered that lane out while still looking activation-ready. Blocking this
item-level drift before proposal review makes skill-route discovery harder to bypass and keeps activation tied to local
validation.

## Change

- Added item-level unsupported-lane detection in `build_route_hint_lane_map`.
- Added `unsupported_lanes`, `lane_status`, and `unsupported_lane_count` to the skill-route local lane candidate surface.
- Made unsupported item-level skill lanes fail route-hint preflight and block the local candidate activation gate.
- Added a regression test that mutates selected skill evidence to include `runtime_execution` and asserts the proposal review is rejected.
- Documented the lane-drift behavior in `docs/skill-route-discovery.md`.

Self-model: unchanged. The existing self-model already says useful growth should prefer locally validated behavior changes and keep external skill routes bounded.
