# Provider Runtime Control Pass 3: Gateway Base URL Preflight

## Evidence

- Source digest: `github-growth-20260623T051652.682999Z`
- Primary evidence URL reviewed: `https://github.com/omnigent-ai/omnigent`
- Reusable pattern extracted: Omnigent documents a multi-harness provider setup surface where gateway base URLs differ by harness family. In particular, OpenRouter-style Claude routes use an Anthropic-compatible `/api` endpoint, while Codex/OpenAI-agent routes use an OpenAI-compatible `/api/v1` endpoint.

## Hypothesis

Provider runtime control should block before harness launch when a gateway base URL endpoint shape is valid for one harness family but mismatched for the selected harness. The diagnostic must remain body-free: labels, hashes, booleans, and recovery hints are acceptable; raw URLs and credential values are not.

## Change

- Added a `gateway_base_url` provider runtime preflight that:
  - infers gateway/harness endpoint expectations for OpenRouter-style routes,
  - classifies configured base URL endpoint shape without exporting the URL,
  - blocks runner invocation on missing, malformed, or harness-mismatched gateway base URLs,
  - emits a supervisor recovery hint with body-free fields only.
- Added a local fixture and focused regression test for an OpenRouter Claude mismatch and Codex match.
