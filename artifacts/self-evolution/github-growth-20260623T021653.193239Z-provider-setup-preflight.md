## Evidence

- https://github.com/omnigent-ai/omnigent/issues/890 showed a setup failure where global `npm install -g` fails with `EACCES` when the global prefix is not user-writable.
- https://github.com/omnigent-ai/omnigent/issues/875 showed a LiteLLM adapter readiness gap where model names can fail unless provider-prefixed and setup should discover/select available models.

## Hypothesis

Provider/setup failures should be visible through the existing local provider-runtime preflight lane before any installer or provider runtime starts. A metadata-only `setup_preflight` block can convert CLI install permission and LiteLLM model-readiness evidence into bounded local diagnostics, recovery hints, and replayable tests.

## Changes

- Added provider setup preflight evaluation to `src/blackhole_agent/harness_eval.py`.
- Added blocker failure modes for unwritable global npm prefixes, missing LiteLLM model prefixes, and missing LiteLLM discovery results.
- Added recovery hints that expose only counts, booleans, and stable codes.
- Added focused regression tests in `tests/test_harness_eval.py`.
- Updated `docs/architecture.md` with the operator-visible contract.

## Self-Model Decision

`docs/self-model.md` was left unchanged. The current file already explicitly allows provider/config preflight checks when rollback-backed and locally validated; this run did not produce evidence that the self-model is behavior-shaping enough to need another revision.
