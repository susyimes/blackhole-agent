## Hypothesis

Skill-route discovery already maps external skill repositories to bounded local lanes, but provider/runtime preflight evidence should be sampled in the same operator-visible path before activation. A blocked or degraded provider/runtime replay sample should hold skill-route promotion and expose body-free recovery hints, rather than allowing a local lane to appear ready while provider/runtime diagnostics are unresolved.

## Changes

- Added optional `provider_runtime_preflight_samples` handling to `skill_route_discovery_lane`.
- Added body-free `provider_runtime_replay_sample` summaries with counts, status, failure modes, hint codes/hashes, supervisor decision, operator next action, and replay commands.
- Blocked skill-route activation with `provider_runtime_replay_not_ready` when a supplied provider/runtime replay sample is blocked or degraded.
- Surfaced the replay sample through activation lanes, local lane intake, and operator handoff without raw URLs, paths, env keys, diagnostics, credentials, provider bodies, or upstream skill bodies.
- Documented the new replay-sampling behavior in `docs/skill-route-discovery.md`.
- Added a regression covering COMPASS/FableCodex/Three.js-style evidence plus a blocked provider-runtime sample.
